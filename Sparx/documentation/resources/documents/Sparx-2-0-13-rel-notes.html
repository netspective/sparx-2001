Sparx 2.0.13 Release Notes

Migrating to Sparx 2.0.13

  1. Thoroughly read the release note contents in the following sections – especially the directory structure changes.
  2. Remove your current SPARX_HOME (e.g C:\Projects\Sparx) directory (or, better yet, back it up first and then remove it)
  3. Checkout a new copy of Sparx from CVS immediately or grab a new eval copy from Netspective when it’s posted on Sep 1.
  4. If you used CVS, go to SPARX_HOME/java directory. Type build and press enter to begin compiling – the new build file will automatically figure out locations of SPARX_HOME and find all the appropriate libraries.
  5. Because the directory structure has changed drastically from earlier builds, it will be best to migrate your application files manually. If the build in step 4 was successful, cd to SPARX_HOME/tools. Type new-sparx-app c:\Projects\xxx where xxx is the name of your application and C:\Projects is the directory where you want to create the app. For example, type new-sparx-app c:\Projects\abc  if your application is called abc. If all goes well, a brand-new blank application will be created.
  6. At the end of step 5 you will be placed into your new application’s WEB-INF directory – c:\Projects\abc\WEB-INF  
  7. Copy c:\Projects\abc\WEB-INF\conf\resin-web.xml to c:\Projects\abc\WEB-INF\web.xml. This is required because the default web.xml that is created by new-sparx-app is setup for Tomcat, not resin.       
  8. Now you need to start copying your other (web-xxx) application’s UI, SQL, CLASSES, JSP, and other files to the new application. Don’t worry if you mess anything up since you can always run new-sparx-app again to create a new shell. DO NOT copy any of the following files over to the new structure:
    1. your old WEB-INF/configuration.xml file -- the new one in WEB-INF/conf/sparx.xml is more recent and more generic
    2. WEB-INF/tld directory contents -- the new ones are more current and generic
    3. WEB-INF/classes/log4j.properties -- the new one is more current and generic
  9. Once you have copied all of your files over, make sure to change the location of your application file in resin.conf to point to your c:\Projects\abc directory. You should also remove the single line referring to the “/shared” context in resin.conf – starting with 2.0.9, Sparx files are no longer shared across apps by default.
  10. This might be a good opportunity to download and install Resin 2.1.4 (latest version) as well.
  11. Once you're all done run "build" from c:\Projects\abc\WEB-INF.

Directory structure has changed to support standard .war file creation and deployment. The format first changed in version 2.0.9 has now been completed in 2.0.13.

  • The major change has been to remove the Site, Documents, and Database directories and make what was in Site now the APP_ROOT.
  • Each application now has a local copy of all Sparx libraries and resources (javascripts, css, xslt, etc). This was required so that a Sparx application can easily become a .war file. The sparx resources that are required by web browsers is under APP_ROOT/sparx. By default, nothing is shared between sparx apps anymore.
  • The build.xml, build.bat, and build.sh files are now in WEB-INF instead of the application root. A new target called “upgrade-sparx” has been added so that when sparx is upgraded, users can just run “build upgrade-sparx” and get the latest copies of all shared files into the current app.
  • The Sparx main config file APP_ROOT/Site/WEB-INF/configuration.xml file is now WEB-INF/conf/sparx.xml
  • The APP_ROOT/Database/schema-src directory is now APP_ROOT/WEB-INF/schema
  • The APP_ROOT/Database/schema directory is now APP_ROOT/WEB-INF/schema/ddl
  • Each application web-xxx is now just called xxx. So, web-hello is now just hello, web-library is just library, etc.
  • Rewrote the build file to be more generic, work with environment variables, and work "out of the box" without requiring users to modify it before compiling. Basically, the batch file now assumes that the directory above it is the SPARX_HOME (which can be overridden using an environment variable).
  • Added a new file called new-sparx-app.bat to SPARX_HOME/tools directory. This batch file can be used to create a brand new “shell” sparx application that can be used as a starter application.
  • All required libraries are now included with Sparx – nothing needs to be downloaded separately.

Other nomenclature changes

  • The default dialog context bean package has been changed from 'dialog.context' to 'app.form.context'.
  • The default DAL package has been changed from 'dal' to 'app.dal'.
  • The generated WEB-INF/lib/dal.jar is now called WEB-INF/lib/appname-dal.jar (e.g. hello-dal.jar, library-dal.jar, cura-dal.jar)
  • The generated WEB-INF/lib/dcb.jar is now called WEB-INF/lib/appname-dcb.jar (e.g. hello-dcb.jar, library-dcb.jar, cura-dcb.jar)

Connection Management

Added the following methods to help centralize connection pooling for all XAF code that does NOT use the ConnectionConext object. This fixes a few problems where connections may not have been freed by XAF SQL DML classes. The following new methods were added (along with JavaDoc comments) to the sparx.xif.db.DatabaseContext class:

                       

            beginConnectionSharing()

            endConnectionSharing()

            getSharedConnection()

                       

Basically these create a better isolation of "shared" connections that are required for running multiple DML statements within a single transaction across a single request.

DAL Data Population

Added ability to ignore or assign null values in the DAL Row classes. Two new methods have been added:

                       

            public void populateDataByNames(DialogContext dc, int valueHandling)

            public void populateDataByNames(DialogContext dc, Map colNameFieldNameMap, int valueHandling)

                       

The valueHandling parameter can be:

VALUEHANDLE_NULLIGNORE or VALUEHANDLE_ASSIGN. NULLIGNORE is used if populateDataByNames should ignore null values altogether; the other is used to assign NULL or zero-length values to columns. The above new methods are called by the following existing methods with the valueHandling parameter set to VALUEHANDLE_NULLIGNORE (same as today so current code is not broken):                 

public void populateDataByNames(DialogContext dc)

            public void populateDataByNames(DialogContext dc, Map colNameFieldNameMap)

Schema

Added "default" to the array of allowed column macro values. Because the DAL generates the default value for inserts and updates when a value is not present, in some cases the value might need to be database specific.  An instance of this is a date colum: in Oracle, the function is "sysdate", while in SQL Server it is "getdate()"  By paramterizing the default value, it is possible to create a truly database-independent tabletype that can then be implemented in a database-specific manner.  Here is an example using the Default tabletype, abstracted one level to allow for this functionality:

                       

       <tabletype name="_Default">

         <param name="sysdatefunc">sysdate</param>

              <param name="record-status-enum">Record_Status</param>

              <column name="cr_stamp" type="stamp" required="dbms" default="$sysdatefunc$" descr="Date/time record was created"/>

              <column name="cr_person_id" lookupref="Person.person_id" descr="The user that created the record"/>

              <column name="cr_org_id" lookupref="Org.org_id" descr="The org that created the record (the one the cr_user_id was logged-into)"/>

              <column name="record_status_id" lookupref="$record-status-enum$" default="0" required="yes" descr="The status of a row in the table"/>

       </tabletype>

      

       <tabletype name="Default" type="_Default">

         <param name="sysdatefunc">getdate()</param>

       </tabletype>

Miscellaneous changes

  • Added new build number management technique. Now the version number x.y.z is managed manually and the build number is automatically incremented each time a build is done.  What this means is that the build number more accurately represents a build.
  • New stylized report and dialog skins made default (better appearance).
  • Updated report skins so that they each have 'xxx' and 'xxx-compressed' versions and updated all skins to use new stylized skins.
  • Created new JSP Panel tag that creates panels that look similar to those created by SQL reports. Sample is visible in hello and library sample apps.
  • Added hyperlinks to Java class names in Dialogs page so that the code can be seen from ACE.
  • Added new file called APP_ROOT/sparx/resources/sparx.build.properties that represents version information for the Sparx build that the web-shared files are related to.
  • Added ability for <xaf:statement> tags to now just execute SQL and produce no report nor store any value ( useful for DML).
  • Cleaned up remaining connection closing issues (where certain connections remained open without being closed).
  • Added a check to ensure that each column is using a datatype that includes an appropriate java class element so that the DAL can be properly generated. Error is now shown in ACE in the SchemaDoc (XML) page.
  • Added a new method called getValueForSqlBind to the DAL Column classes so that classes like java.util.Date, which need to stored in JDBC bind parameters as java.sql.Date, can automatically take care of the conversion in DAL.
  • Added a new test database context that does not require JNDI.
  • Added server-side checks for new field.xxx min-length attribute and existing max-length attribute.
  • Added new field type called <field.html> that allows arbitrary HTML to be included in dialogs. This field should be used in the place of <field.static> when buttons and general HTML needs to be included. The actual HTML is enclosed inside the <field.html> tag body. For this field, the "name" parameter is optional. Usage: <field.html> ... arbitrary html </field.html> or <field.html caption="test"> html </field.html>.
