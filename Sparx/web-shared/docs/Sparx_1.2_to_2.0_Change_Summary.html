Sparx 2.0 Structural Changes

  • CVS module is now called Sparx instead of Framework. The Release 1.2.8 branch will remain in Framework but all new 2.0 work will happen in Sparx.
  • The JAR name has changed from xaf-1_2_8.jar to sparx-2-0-1.jar.
  • Optimized and expanded all imports (all imports are clearly marked now, without wildcards) and cleaned up tabulation problems in the source code.
  • Refactored many packages to more closely correlate with the nomenclature used in the online documentation – some of the significant refactorings are:
    • com.xaf.* is now com.netspective.sparx.*
    • com.xaf.ace is now com.netspective.sparx.ace
    • com.xaf.ant is now com.netspective.sparx.util.ant
    • com.xaf.db.schema is now com.netspective.sparx.xif.dal
    • com.xaf.db.SchemaDoc* is now in com.netspective.sparx.xif
    • com.xaf.form.taglib.*, com.xaf.navigate.taglib.*, com.xaf.task.*Tag moved to com.netspective.sparx.xaf.taglib (basically, all JSP tags now reside under one directory)
    • com.xaf.query is now com.netspective.sparx.xaf.querydefn
    • com.xaf.xml is now com.netspective.sparx.xml

Content Changes

  • Copyright notices have been added to all the source files
  • In configuration.xml, all entries that began with framework.* are now called sparx.*.
  • Dialog/form listeners have been removed (they caused too many problems and poor code). Dialog subclassing is now the preferred method of adding Java code to Dialogs.
  • The xaf.tld (taglib descriptor) file is now called sparx.tld.

API Changes

  • In com.netspective.sparx.xaf.config.Configuration.java, the String getValue(String) method is now called String getTextValue(String) and will now produce an error if the property does not exist.
  • In com.netspective.sparx.form.Dialog.java (and all sublasses), the String getHtml(DialogContext) method has been replaced with void renderHtml(Writer, DialogContext) method. This new method signature now improves performance by writing dialogs directly to the output stream provided instead of storing the dialog HTML in a string and then writing out that string. This is a major change to the way dialogs are rendered and it will break your current WEB-INF/classes/…/AppLoginDialog.java file and PageTag.java file. It is very simple to change the API and recompile, though.
  • In com.netspective.sparx.xaf.form.DialogField.java (and all subclasses), the String getControlHtml(DialogContext) method has been replaced with void renderControlHtml(Writer, DialogContext) method. This new method signature now improves performance by writing dialogs directly to the output stream provided instead of storing the field HTML in a string and then writing out that string. This is a major change to the way dialog fields are rendered, but it shouldn’t break any existing code.
  • In com.netspective.sparx.security.LoginDialog.java (and all subclasses) the void producePage(DialogContext, Writer) method signature has been changed to void producePage(Writer, DialogContext) to be more consistent with the changes mentioned above. This minor change will break your current WEB-INF/classes/…/AppLoginDialog.java. It is very simple to change the calling sequence and recompile, though.

How to Migrate from 1.2.8 to 2.0.1

To help with the migration process and do some of the variable replacements, a new target called migrate-xaf12-sparx20 is provided in the application’s ANT build.

 

  1. Backup your current project because many of the files will change in this process. Don’t continue without doing a backup.
  2. Using the same CVSROOT as before, checkout the new module called Sparx. (cvs checkout Sparx). Remember, the old Framework directory is now going to be called Sparx. This new Sparx directory (wherever you put it) will be known as the SPARX_HOME.
  3. Update your Tomcat or Resin conf files to point the /shared/ directory to this new SPARX_HOME/web-shared instead of the old Framework/web-shared.
  4. CD to the application root (e.g. web-cura) and make sure that the build.bat script in your application’s main directory (web-cura, web-xxx, etc) has SPARX_HOME and APP_ROOT setup to the proper directories. If you need an example of the build.bat file’s content, use the sample provided in SPARX_HOME/tools/app-build.bat or app-build.sh.
  5. Run build.bat or build.sh with the target ‘migrate-xaf12-sparx20’ like
          
    build migrate-xaf12-sparx20

    The above script will replace your existing configuration.xml file and tld/xaf.tld and install fresh copies of the files. It will make variable replacements in the new configuration.xml file and will replace the old package and class names with the new ones in all the files in your project. To see everything that it does, please refer to SPARX_HOME/tools/app-build.xml and look at the ‘migrate-xaf12-sparx20’ target in that file.
  6. In WEB-INF/classes, find AppLoginDialog.java and change the line
    from: public void producePage(DialogContext dc, Writer writer) throws IOException
    to: public void producePage(Writer writer, DialogContext dc) throws IOException
  7. In the same file, AppLoginDialog, same method producePage, change line
    from: writer.write(getHtml(dc, true));
    to: renderHtml(writer, dc, true);
  8. In WEB-INF/classes, find PageTag.java and change the line in the method doLogin
    from: loginDialog.producePage(dc, resp.getWriter());
    to: loginDialog.producePage(resp.getWriter(), dc);
  9. Once all the steps above have been completed, run build in the main application directory (web-cura, etc).
